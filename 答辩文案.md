## 二、研究背景与意义（约2分钟）

### 2.1 现实背景

当前高校考试管理存在以下几个突出问题：

1. **题库建设效率低**：教师手工录入题目耗时耗力，题目质量参差不齐；
2. **组卷方式落后**：依赖人工经验组卷，难以保证试卷的知识点覆盖率和难度均衡性；
3. **试卷重复率高**：不同年份、不同教师出的试卷存在大量重复题目，考试公平性受到影响；
4. **权限管理混乱**：多角色（教师、教研室主任、教务处）协同管理缺乏规范的权限控制；
5. **缺乏智能辅助**：传统系统没有AI辅助功能，无法自动生成题目或智能审题。

### 2.2 研究意义

本系统的研究与实现具有以下重要意义：

- **提升出题效率**：通过AI大模型自动生成试题，将教师出题时间缩短70%以上；
- **保障考试质量**：多维度统计分析（难度分布、知识点覆盖、认知层次）确保试卷质量；
- **防止试题泄露**：内置试卷查重系统，实时检测A/B卷重复率；
- **规范管理流程**：基于角色的权限控制，实现从出题→审题→组卷→考试的全流程规范化管理。

## 三、系统总体设计（约3分钟）

### 3.1 系统架构

本系统采用经典的 **Java EE 三层架构**，结合 **Spring MVC + MyBatis** 框架体系构建：

```

│                   表现层（View）                      │
│         JSP + jQuery + UEditor富文本编辑器            │
├─────────────────────────────────────────────────────┤
│                   控制层（Controller）                │
│              Spring MVC Controller                   │
├─────────────────────────────────────────────────────┤
│                   业务层（Service）                   │
│  QuestionService / PaperService / CourseService /    │
│  UserService / IntelliQuestionService / VerifyService│
├─────────────────────────────────────────────────────┤
│                   数据访问层（DAO）                   │
│              MyBatis + Oracle 数据库                  │
├─────────────────────────────────────────────────────┤
│                   基础设施层                          │
│  Apache Shiro（安全） / EhCache（缓存） /             │
│  Druid（连接池） / Log4j2（日志）                     │
└─────────────────────────────────────────────────────┘
```

**外部集成服务：**
- 百度文心大模型（ERNIE-4.5-turbo-32k）：AI智能出题
- 讯飞TTS语音合成服务：题目语音朗读
- 试卷查重服务（独立微服务）：A/B卷重复率检测

### 3.2 数据库设计

系统使用 **Oracle** 数据库，核心数据表包括：

| 模块 | 核心表 | 说明 |

系统采用 **Apache Shiro** 实现基于角色的访问控制（RBAC），支持以下角色：

## 四、核心功能模块详解（约6分钟）

### 4.1 课程与题库管理模块

**功能描述：**

课程是整个系统的核心组织单元。每门课程下包含：
- **三级知识点大纲（Theme）**：支持树形结构，最多三级，用于题目的知识点归类
- **题型配置**：单选题、多选题、判断题、填空题、简答题、案例分析题等多种题型
- **难度等级**：支持自定义难度分级
- **认知层次**：记忆、理解、应用等认知分类（对应布鲁姆教育目标分类）
- **知识点分布**：支持多维度知识点标注

**技术实现：**

在 [`CourseService.java`](src/com/cx/kaoyi/business/service/CourseService.java) 中，通过 `getThemeTree()` 方法实现知识点树形结构的递归查询，支持按父节点ID检索子节点，构建完整的三级大纲树。

### 4.2 试题管理模块

**功能描述：**

试题管理是系统的核心模块，支持：

1. **手动录入**：通过 UEditor 富文本编辑器录入题目，支持数学公式（MathML/LaTeX）、图片、表格等富媒体内容
2. **Word批量导入**：上传Word文档，系统自动解析题目内容，支持公式图片转换（EMF/WMF/TIFF格式自动转PNG/SVG）
3. **AI智能生成**：调用百度文心大模型自动生成题目（详见4.5节）
4. **题目审核**：支持一审、二审流程，审核通过后题目才能用于组卷
5. **题目查重**：检测题库内重复题目，防止题目冗余
6. **工作量统计**：统计每位教师的出题数量、审题数量

**技术亮点——Word公式解析：**

在 [`IntelliQuestionService.java`](src/com/cx/kaoyi/business/service/serviceImpl/IntelliQuestionService.java) 的 `getImportWordStringWithFormula()` 方法中，系统实现了复杂的Word文档解析逻辑：
- 使用 Apache POI 解析 `.docx` 文件
- 识别 `m:oMath` 节点（Office数学公式），调用 LaTeX 转换器生成 SVG 图片
- 处理 EMF、WMF、TIFF 等特殊图片格式，通过 GraphicsMagick 转换为标准 PNG
- 将图片同步到 Nginx 静态资源目录，确保前端可访问

### 4.3 智能组卷模块

**功能描述：**

系统提供多种组卷方式：

| 组卷方式 | 说明 |
|----------|------|
| 手动组卷 | 教师手动从题库中选题 |
| 随机组卷 | 按题型、难度、知识点参数随机抽题 |
| 难度自适应组卷 | 根据目标难度系数自动调整题目组合 |
| 多课程联合组卷 | 支持跨课程组卷 |
| 合并组卷 | 从多份已有试卷中合并生成新试卷 |
| A/B卷生成 | 自动生成内容相同但顺序不同的A/B卷 |

**试卷质量预览：**

组卷完成后，系统提供试卷质量分析面板，展示：
- 难度分布饼图
- 知识点覆盖率
- 认知层次分布
- 题型分布统计
- 总分与答题时间估算

**技术实现：**

在 [`PaperService.java`](src/com/cx/kaoyi/business/service/PaperService.java) 中，`getRandomQuestion()` 方法实现了按多维度参数的随机抽题逻辑，`generateBpaper()` 方法实现了B卷的自动生成（打乱题目顺序和选项顺序）。

### 4.4 试卷导出模块

**功能描述：**

系统支持将试卷导出为标准 Word 文档，支持多种格式模板：

- **标准格式**（seePaper.jsp）
- **重庆格式**（seePaper-chongqing.jsp）：使用 FreeMarker 模板引擎生成符合重庆地区规范的试卷格式
- **云南格式**（seePaper-yunnancj.jsp）：符合云南地区考试规范
- **海军格式**（seePaper-haijun.jsp）：符合军队院校考试规范

**技术实现：**

在 [`PaperExporter.java`](src/com/cx/kaoyi/framework/handler/PaperExporter.java) 中，系统使用 **FreeMarker 模板引擎** 结合 **Apache POI** 生成 Word 文档。对于包含数学公式的题目，通过 **Aspose.Words** 库进行高质量的公式渲染，确保导出的 Word 文档中公式显示正确。

### 4.5 AI智能出题模块（核心创新）

这是本系统最具创新性的功能模块，也是本项目的核心亮点。

**功能描述：**

教师只需选择课程、知识点大纲、题型，系统即可自动调用 **百度文心大模型（ERNIE-4.5-turbo-32k）** 生成高质量试题，并存入AI题库暂存区，经教师审核后可一键导入正式题库。

**工作流程：**

```
教师设置参数
（课程+知识点+题型+数量）
        ↓
系统构建Prompt
（包含课程名、大纲主题词、题型要求、答案格式）
        ↓
调用百度文心大模型API
（ERNIE-4.5-turbo-32k，开启联网搜索）
        ↓
AI返回JSON格式题目
（题目内容+选项+答案+解析）
        ↓
系统解析并存入AI题库暂存区
        ↓
教师审核 → 一键导入正式题库
```

**技术实现细节：**

1. **Prompt工程设计**（[`QuestionGeneratorCharger.java`](src/com/cx/kaoyi/framework/GPT/utils/quesitonGenerate/QuestionGeneratorCharger.java)）：
   - 系统为不同题型设计了专门的Prompt模板（存储在 `resources/gptfiles/` 目录下）
   - 支持15种题型格式：单选题（question0.txt）、多选题（question1.txt）、判断题（question2.txt）、主观题（question3.txt）、案例分析题（question10-14.txt）等
   - 开启百度文心的**联网搜索功能**（WebSearch），确保生成题目的时效性和准确性
   - 通过**二次AI调用**细化主题词，增加题目多样性（`findQuestionTheme()` 方法）

2. **异步多线程生成**（[`GenerateAsyncTask.java`](src/com/cx/kaoyi/framework/GPT/generateDTO/business/GenerateAsyncTask.java)）：
   - 使用 `QuestionGenerateThreadPool` 线程池管理生成任务
   - 每个课程的生成任务独立运行，互不干扰
   - 支持案例分析题（iscon=1）的特殊处理：先生成题干，再生成子题

3. **容错与重试机制**：
   - 网络超时自动重试（最多1次）
   - JSON解析失败自动重试
   - 支持英文题目生成（检测主题词是否为纯英文）

4. **AI智能编辑功能**（[`QuestionOperatorCharger.java`](src/com/cx/kaoyi/framework/GPT/utils/operate/QuestionOperatorCharger.java)）：
   - 支持对已有题目进行AI辅助修改
   - 支持AI辅助生成题目解析
   - 实时流式输出，提升用户体验

### 4.6 试卷查重模块

**功能描述：**

系统集成了独立的试卷查重微服务，实现：
- **A/B卷重复率检测**：计算A卷和B卷之间的题目重复比例
- **历史试卷查重**：检测当前试卷与历史试卷的重复情况
- **单题查重**：查看某道题目在其他试卷中的出现情况
- **PDF存档**：将试卷HTML内容上传至查重服务器，生成PDF存档

**技术实现：**

在 [`StatsApiClient.java`](src/com/cx/kaoyi/framework/question/repeat/StatsApiClient.java) 中，系统通过 OkHttp 客户端与查重微服务进行 RESTful API 通信，使用 API Key 进行鉴权，支持以下接口：
- `checkRepeatInfo`：获取A/B卷重复率
- `queryDedupStatsMapByExam`：获取试卷内所有题目的查重统计
- `statsDetailByQid`：获取单题的重复详情
- `uploadHtmlAsPdf`：上传试卷HTML生成PDF存档

### 4.7 讯飞TTS语音合成模块

**功能描述：**

系统集成了**科大讯飞TTS（Text-to-Speech）语音合成**服务，支持将题目内容转换为语音，主要用于：
- 英语听力题的语音生成
- 无障碍考试支持

**技术实现：**

在 [`WebTtsWsService.java`](src/com/cx/kaoyi/framework/xunfei/utils/WebTtsWsService.java) 中，系统通过 **WebSocket** 协议与讯飞TTS API 进行实时通信：
- 使用 **HMAC-SHA256** 算法生成鉴权签名
- 支持多种发音人（VoiceRole）、语速（speed）、音调（pitch）配置
- 采用 **CompletableFuture** 异步接收音频数据
- 音频格式为 MP3（lame编码）

### 4.8 在线考试模块

**功能描述：**

系统支持学生在线参加考试：
- 考试倒计时与自动交卷
- 防作弊机制（考试锁定管理 `ExamLockManager`）
- 考试结果分析（`analysis.jsp`）
- 教师在线阅卷（`remark2s.jsp`）
- 学生查看试卷（`viewStudentPaper.jsp`）

## 五、安全设计（约1分钟）

### 5.1 认证与授权

系统使用 **Apache Shiro** 框架实现安全控制：

- **验证码登录**（[`KaptchaFormAuthenticationFilter.java`](src/com/cx/kaoyi/framework/shiro/KaptchaFormAuthenticationFilter.java)）：登录时需输入图形验证码，防止暴力破解
- **密码加密**：使用 **SHA-256 + Salt + 3次迭代** 的哈希算法存储密码，防止彩虹表攻击
- **密码错误锁定**：记录密码错误次数（`PasswordWrongRecord`），超过阈值锁定账户
- **Session管理**：自定义 `ShiroSessionManager`，支持分布式Session（`LocalSessionDAO`）
- **多端登录支持**：支持PC端和移动端（`UserAgentUtils`）的不同登录策略

### 5.2 数据安全

- **AES加密**（[`AES.java`](src/com/cx/kaoyi/framework/utils/AES.java)）：对敏感数据进行AES加密传输
- **XSS防护**：使用 OWASP Java Encoder 对输出内容进行编码，防止XSS攻击
- **SQL注入防护**：MyBatis 参数化查询，杜绝SQL注入
- **IP访问控制**（`IPRange.java`）：支持IP白名单/黑名单配置

## 九、常见答辩问题及参考回答

### Q1：为什么选择百度文心大模型而不是ChatGPT？

**回答：**
选择百度文心大模型（ERNIE-4.5-turbo-32k）主要基于以下考虑：
1. **合规性**：国内高校项目需要使用合规的国产AI服务，百度文心是经过国家备案的大模型服务；
2. **中文能力**：文心大模型对中文专业术语的理解和生成能力更强，特别适合中文医学、法律等专业题目的生成；
3. **联网搜索**：文心大模型支持实时联网搜索，能生成包含最新知识的题目；
4. **API稳定性**：百度千帆平台在国内访问稳定，不存在网络访问问题；
5. **成本**：相比OpenAI，百度文心的API调用成本更低，适合高校预算。

### Q2：AI生成的题目质量如何保证？

**回答：**
系统通过以下机制保证AI生成题目的质量：
1. **精准Prompt设计**：为每种题型设计了专门的Prompt模板，明确指定题目格式、难度要求和专业领域；
2. **二次主题词细化**：在生成题目前，先调用AI对主题词进行细化分析，从多个角度出题，增加题目多样性；
3. **人工审核机制**：AI生成的题目存入暂存区，必须经过教师审核后才能进入正式题库，确保质量把关；
4. **容错重试**：JSON解析失败或网络超时时自动重试，确保生成成功率；
5. **格式规范化**：系统对AI返回的JSON进行严格解析和校验，确保题目格式符合系统要求。

### Q3：系统如何处理数学公式？

**回答：**
系统对数学公式的处理分为两个场景：
1. **Word导入时**：使用Apache POI解析Word文档中的`m:oMath`节点（Office数学公式），通过XSLT转换器将其转换为LaTeX格式，再调用Node.js的MathJax库渲染为SVG图片，存储在服务器上；
2. **在线编辑时**：集成了UEditor富文本编辑器，支持直接插入MathML格式的数学公式，在前端通过MathJax进行渲染显示；
3. **导出Word时**：使用Aspose.Words库将HTML中的数学公式转换为Word原生公式格式，确保导出文档中公式显示正确。

### Q4：系统的并发性能如何？

**回答：**
系统在并发性能方面做了以下优化：
1. **数据库连接池**：使用Druid连接池，配置初始连接数100，最大活跃连接220，最小空闲连接50，能够支撑高并发访问；
2. **AI生成线程池**：使用`QuestionGenerateThreadPool`管理AI生成任务，每个课程的生成任务独立运行，互不阻塞；
3. **Spring异步任务**：配置了10个线程的异步执行器（`asyncExecutor`），处理耗时操作；
4. **Session管理**：自定义`LocalSessionDAO`实现Session的本地缓存，减少数据库Session查询压力；
5. **分页查询**：使用PageHelper分页插件，避免全表扫描，提升查询性能。

### Q5：系统的安全性如何保证？

**回答：**
系统从多个层面保证安全性：
1. **认证安全**：登录需要图形验证码（Kaptcha），防止暴力破解；密码使用SHA-256+Salt+3次迭代哈希存储；
2. **授权安全**：基于Apache Shiro的RBAC权限控制，每个操作都需要对应权限；
3. **传输安全**：敏感数据使用AES加密传输；
4. **注入防护**：MyBatis参数化查询防止SQL注入；OWASP Encoder防止XSS攻击；
5. **访问控制**：支持IP白名单/黑名单配置，限制非法IP访问；
6. **考试安全**：`ExamLockManager`实现考试锁定机制，防止考试过程中的异常操作。

### Q6：系统支持哪些题型？

**回答：**
系统支持以下题型（通过`atid`字段区分）：
- **客观题**：单选题（atid=0/2/8）、多选题（atid=1/3/9）、判断题（atid=4）
- **主观题**：填空题（atid=5）、简答题（atid=6）、论述题（atid=7）、计算题（atid=10）、综合题（atid=11）
- **特殊题型**：英语听力题（atid=12）、在线编程题（atid=13）
- **案例分析题**（iscon=1）：包含题干和多个子题的复合题型，如B1型题（医学专用）

### Q7：系统如何实现试卷查重？

**回答：**
试卷查重功能通过独立的查重微服务实现，主要流程如下：
1. **数据同步**：当试卷被操作（添加/修改题目）时，系统通过`PaperChangeRecorder`记录变更，并将题目数据同步到查重服务器；
2. **相似度计算**：查重服务器对题目内容进行文本相似度计算，生成题目的哈希指纹（`Hash128`）；
3. **重复率查询**：通过`StatsApiClient`调用查重服务的REST API，获取A/B卷重复率（`RepeatABRate`）和具体重复题目对（`DuplicatePairDTO`）；
4. **结果展示**：在试卷管理界面展示重复率百分比，并支持查看具体重复题目的对比详情；
5. **PDF存档**：将试卷HTML上传至查重服务器生成PDF，用于后续的人工核查。

### Q8：系统的数据库为什么选择Oracle而不是MySQL？

**回答：**
选择Oracle数据库主要基于以下原因：
1. **企业级需求**：高校考试系统对数据可靠性要求极高，Oracle在事务处理、数据一致性方面更有保障；
2. **大数据量支持**：题库可能包含数十万道题目，Oracle在大数据量查询和索引优化方面性能更优；
3. **存储过程支持**：系统使用了Oracle存储过程（如`call_updateCourseQuestioncount`）来维护课程题目数量统计，Oracle的存储过程功能更强大；
4. **CLOB类型支持**：题目内容可能包含大量富文本HTML，Oracle的CLOB类型（通过`OracleClobTypeHandler`处理）能更好地存储大文本；
5. **现有基础设施**：学校已有Oracle数据库基础设施，复用现有资源降低部署成本。

### Q9：如果让你重新设计，你会做哪些改进？

**回答：**
如果重新设计，我会考虑以下改进：
1. **微服务化**：将题库管理、试卷管理、AI生成、查重等模块拆分为独立微服务，提升系统可扩展性；
2. **前后端分离**：采用Vue.js或React替代JSP，提升前端用户体验和开发效率；
3. **引入向量数据库**：使用Milvus等向量数据库存储题目的语义向量，实现基于语义相似度的智能查重，比传统文本匹配更准确；
4. **RAG增强**：将学校的教材和课件作为知识库，通过RAG（检索增强生成）技术让AI生成的题目更贴近教学内容；
5. **数据库迁移**：考虑迁移到PostgreSQL，降低授权成本，同时PostgreSQL的JSON支持和全文检索功能也很强大。
